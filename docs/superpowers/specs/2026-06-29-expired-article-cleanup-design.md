# 过期文章清理优化设计

## 背景

当前 `DELETE /api/v1/wx/articles/clean_expired` 在请求内完成全部工作：

1. 查询所有七天前的文章。
2. 逐篇查询 `article_images`。
3. 逐张串行调用 Supabase Storage 删除对象。
4. 再次查询过期文章并批量删除数据库记录。

Nginx 默认等待上游响应 60 秒。超时后浏览器收到 504，但后端协程仍继续运行；用户再次点击会产生多个重叠任务。

## 目标

- 清理请求在正常 API 延迟内返回，不再依赖增大 Nginx 超时。
- 同一进程同一时间只运行一个过期文章清理任务。
- 前端可以查询任务状态并在完成后刷新文章列表。
- 消除逐篇图片映射查询和逐张串行 Storage 删除。
- 单个 Storage 对象删除失败不应中止整个清理任务，但必须计数和记录。
- 保留现有“删除七天前文章”的业务语义。

## 非目标

- 不新增持久化任务表、数据库 RPC 或跨实例分布式锁。
- 不改造“清理孤儿文章”“清理重复文章”和手工删除接口。
- 不通过调大 Nginx 超时掩盖同步清理耗时。

## 方案选择

### 采用：进程内后台任务

后台服务持有当前任务和状态快照。启动接口创建 `asyncio.Task` 后立即返回；重复启动返回现有任务。状态接口返回 `idle`、`processing`、`success` 或 `failed`。

该方案匹配当前单后端容器、单应用进程的部署方式，改动小且能直接解决故障。

### 未采用：Supabase 持久任务队列

持久队列支持进程重启恢复和多实例互斥，但需要新表、claim RPC、worker 生命周期和迁移。当前部署没有这项复杂度需求。

### 未采用：仅优化同步接口

批量化可以显著缩短耗时，但数据量和外部 Storage 延迟仍可能突破代理超时，且无法阻止超时后的重复执行。

## 后端设计

新增独立的过期文章清理服务，负责：

- `start_expired_cleanup(days=7)`：若已有任务运行，返回同一任务并标记 `already_running=true`；否则生成 `run_id`、初始化状态并创建后台任务。
- `get_expired_cleanup_status()`：返回状态快照，不暴露可变内部对象。
- 后台执行器：更新开始、进度、完成和失败状态；使用 `logger.exception` 保留异常类型和堆栈。

状态字段：

- `run_id`
- `status`: `idle | processing | success | failed`
- `already_running`
- `matched_count`
- `deleted_count`
- `storage_deleted_count`
- `storage_failed_count`
- `started_at`
- `finished_at`
- `error`

API 调整：

- `DELETE /api/v1/wx/articles/clean_expired` 返回 HTTP 202 和任务状态。
- 新增 `GET /api/v1/wx/articles/clean_expired/status`。

## 清理数据流

1. 一次查询过期文章，只读取 `id,content`。
2. 将文章 ID 分块，一次查询一批 `article_images`，建立 `article_id -> object_path[]` 映射。
3. 没有映射的历史文章才从 HTML 正文提取 Storage 路径。
4. 对去重后的路径执行有上限的并发删除。一个批次复用同一个 `httpx.AsyncClient`；每个对象独立捕获异常并计入失败数。
5. 将文章 ID 分块删除，避免超长 `in (...)` 请求。
6. 映射表继续依赖现有外键级联删除；服务不再逐文章清理映射。

默认并发上限设为 8，兼顾吞吐和 Supabase 压力。分块大小设为 100，避免 URL 和 PostgREST 参数过长。

Storage 删除保持 best-effort 语义：对象删除失败会记录，但不会阻止数据库文章删除。任务成功响应中包含失败计数，便于后续观察孤立对象。

## 前端设计

- 点击“清理过期”后，DELETE 请求只负责启动任务。
- 收到任务状态后，每两秒查询状态接口。
- `processing` 时保持清理操作禁用并显示进行中消息。
- `success` 时显示删除数量和 Storage 失败数量，并刷新文章列表。
- `failed` 时显示后端错误摘要并停止轮询。
- 页面重新加载时查询一次状态；如果任务仍在运行则恢复轮询。

## 并发与生命周期

- 单个进程使用任务引用和异步锁保证原子启动。
- Nginx 客户端断开不会影响后台任务。
- 应用关闭时不专门等待清理任务；容器重启会丢失进程内状态，未完成任务可由管理员重新启动。
- 该限制适用于当前单实例部署。若后续增加多个后端副本，必须迁移到持久任务表和数据库 claim 锁。

## 测试

后端测试覆盖：

- 启动接口立即返回 processing 状态。
- 重复启动复用同一个 `run_id`，只创建一个任务。
- 图片映射按批次查询，不发生逐文章查询。
- Storage 删除并发不超过配置上限。
- 单对象删除失败被计数，其他对象和数据库清理继续执行。
- 成功和异常状态正确落为终态。
- API 返回 202，状态接口返回当前快照。

前端通过 TypeScript 构建和现有测试验证类型与行为；轮询终态逻辑提取为纯函数时补充单元测试。

## 验收标准

- DELETE 启动接口不等待实际清理完成。
- 同一时间的重复请求不会创建重叠清理任务。
- 清理实现没有逐篇图片映射查询或逐张串行 Storage 删除。
- 前端能展示进行中、成功和失败状态。
- 后端相关测试、后端完整测试和前端构建均通过。
