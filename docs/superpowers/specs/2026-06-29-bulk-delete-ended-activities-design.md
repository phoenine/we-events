# 批量删除已结束活动设计

## 目标

活动页面提供一次性清理数据库中全部已结束活动的能力。所有已登录用户均可执行，权限策略与当前文章清理功能保持一致。

## 范围

- 删除条件与活动页面一致：根据 `start_at`、`end_at` 实时计算后为 `ended`。
- 删除范围为数据库中的全部匹配记录，不受前端当前加载上限影响。
- 保留现有单条活动删除能力。
- 不引入通用条件删除接口，不调整文章清理权限。

## 后端

新增专用活动清理接口：

```text
DELETE /activities/ended
```

接口继续使用现有登录用户依赖，不限制管理员角色。接口分页扫描全部活动，复用页面列表的状态计算规则筛出已结束记录，再由仓储层按 ID 分批删除，并以实际删除结果计算数量。成功响应返回：

```json
{
  "message": "已删除全部已结束活动",
  "deleted_count": 3
}
```

没有匹配记录时接口仍成功，`deleted_count` 为 `0`。数据库或服务异常沿用活动 API 现有错误响应形式。

## 前端交互

活动列表卡片顶部增加与文章页清理操作一致的工具栏，并放置危险操作按钮“清理已结束”。

- 按钮始终可用，仅在清理请求执行期间禁用。页面只加载部分活动，不能据此判断数据库中是否存在已结束记录。
- 点击后显示确认框：“将删除数据库中全部已结束活动，且不可恢复，是否继续？”
- 确认按钮使用危险样式，执行期间按钮显示加载状态并禁止重复提交。
- 成功后提示“已删除 N 个活动”，关闭当前打开且状态为 `ended` 的详情抽屉，并刷新活动列表。
- 失败时保留当前列表和详情状态，并显示后端错误信息。

后端始终以数据库中的全部活动为范围；没有匹配记录时成功提示删除 `0` 个活动。

## 测试

- 后端 API 测试验证接口要求登录、按页面相同规则重新计算状态，并返回实际删除数量。
- 仓储测试验证删除操作只使用后端计算得到的活动 ID，不接收前端删除条件。
- 前端 API 测试验证请求方法和路径。
- 前端活动页测试验证按钮文案、确认提示、请求期间禁用，以及成功后的列表刷新和详情关闭行为。

## 非目标

- 不提供按勾选项批量删除活动。
- 不允许客户端传入任意状态作为删除条件。
- 不增加软删除、撤销或回收站。
- 不修改现有活动状态计算规则。
