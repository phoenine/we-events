# 前端乐观删除设计

## 目标

文章、活动和公众号在用户确认删除后立即从当前界面消失，后端删除请求在后台继续执行。请求失败时恢复被删除的数据并明确提示用户，最终重新拉取服务端数据以保持一致。

## 范围

本次覆盖以下删除入口：

- 单篇文章删除
- 批量文章删除
- 单个活动删除，包括列表卡片和详情抽屉入口
- 单个公众号删除
- 无效文章清理
- 重复文章清理
- 过期文章清理

公众号分组删除不在本次范围内。

## 方案

继续使用现有 React Query mutation，在每个删除 mutation 中实现标准乐观更新生命周期：

1. `onMutate` 取消对应列表的在途查询，防止旧响应覆盖乐观状态。
2. 保存该资源所有相关列表查询的缓存快照。
3. 从缓存列表中移除目标记录，并同步减少分页总数；批量删除按实际命中的记录数调整总数。
4. 立即清理关联的本地界面状态，例如文章选中项和已打开的活动详情。
5. 后端请求成功后显示成功消息，不再等待重新查询才更新界面。
6. 后端请求失败后恢复本次 mutation 保存的缓存快照，并显示失败消息。
7. `onSettled` 使对应列表查询失效，由服务端结果完成最终校准。

不修改现有后端接口，也不引入软删除、撤销队列或持久化任务。

三类文章智能清理由后端基于全量数据判断，前端缓存无法可靠确定待删除 ID，因此不执行推测性的乐观移除：

1. 用户确认后立即关闭确认弹窗。
2. 三个清理入口统一进入“清理中”状态，避免重复提交。
3. 后端完成后显示结果并使文章列表查询失效。
4. 后端失败时保留当前列表并显示错误提示。

这种处理提供即时操作反馈，同时避免前端复制后端的孤儿、重复和过期判定规则。若未来要求智能清理也立即移除记录，应先增加返回精确候选 ID 的清理预览接口。

## 缓存与分页

文章和公众号列表使用带分页、筛选参数的查询 key。乐观删除更新该资源 key 前缀下的全部现存列表缓存，确保用户切换已缓存分页或筛选条件时不会看到刚删除的数据。

- `ApiList<T>.list` 中移除匹配 ID。
- `ApiList<T>.total` 仅按当前缓存中实际移除的记录数递减，且不低于零。
- 活动列表是数组缓存，直接过滤匹配 ID。
- 批量文章删除后立即清空已删除 ID 对应的选中项。

## 并发与回滚

每次 mutation 只恢复自己开始前捕获的缓存快照。为避免两个并发删除失败回滚时相互覆盖，删除 mutation 按资源类型串行提交：已有删除请求未结束时，对应删除入口保持不可再次提交。不同资源之间可以并行删除。

活动从详情抽屉删除时立即关闭抽屉；若请求失败，列表数据恢复，但抽屉不自动重新打开，避免突然打断用户当前操作。

## 错误反馈

- 成功：保留现有“文章已删除”“活动已删除”“公众号已删除”等成功提示。
- 失败：显示接口错误信息；无法识别时使用对应的“删除失败”默认文案。
- 失败回滚后仍执行查询失效，以服务端状态为最终依据。
- 智能清理：确认后立即显示“清理中”；成功显示后端清理结果，失败显示对应错误。

## 测试

将缓存数据变换提取为小型纯函数，使用现有前端测试方式覆盖：

- 单条删除立即移除记录并减少总数。
- 删除不存在的 ID 不改变总数。
- 批量删除只按实际命中数量减少总数。
- 活动数组删除只移除目标记录。
- mutation 失败时缓存快照可完整恢复。
- 智能清理提交后立即进入处理中状态，且不会推测性修改文章缓存。

最后运行前端测试和生产构建，确认类型检查与打包通过。

## 验收标准

- 用户确认后，目标记录无需等待网络响应即从界面消失。
- 删除失败时记录重新出现，并显示错误提示。
- 删除成功后列表与服务端数据一致。
- 分页总数、批量选中状态和活动详情状态正确。
- 同一资源的重复或并发删除不会导致错误回滚。
- 三类智能清理确认后立即提供处理中反馈，执行期间无法重复提交，完成后列表与服务端一致。
